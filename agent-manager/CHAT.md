# Chat Execution Flow Architecture

This document describes the flow of a chat request from the API entry point through the Service layer and down to the specific LLM integration via Spring AI.

## 1. API Entry Point (`AgentsController`)

The chat request enters the system via the `AgentsController`, which exposes REST endpoints for both synchronous and streaming interactions.

- **Endpoints**:
  - `POST /api/agents/{agentId}/runs` (Blocking execution)
  - `POST /api/agents/{agentId}/runs/stream` (Server-Sent Events streaming)
- **Request Payload (`RunRequest`)**:
  - `message`: The user's prompt text.
  - `session_id`: (Optional) Identifier for conversation history context. If omitted, a new UUID is generated.
  - `media`: (Optional) List of base64 encoded images or URLs for multimodal input.
- **Responsibility**:
  - Extracts parameters from the JSON body.
  - Delegates execution to `AgentService`.
  - Maps the service response (`RunResponse` or `Flux<AgentStreamEvent>`) to the appropriate HTTP response format.

## 2. Service Logic (`AgentService`)

The core orchestration happens within `AgentService`, which bridges the gap between the static agent definition and the dynamic runtime execution.

### A. Agent Lookup
The service first retrieves the agent's configuration using the provided `agentId`:
```java
AgentDefinition def = agentRegistry.findById(agentId);
```
If the agent is a `ROUTER` team, the request is recursively routed to the appropriate sub-agent before proceeding.

### B. ChatClient Construction (`buildChatClient`)
For every request, a fresh `ChatClient` instance is constructed to encapsulate the specific context of the run. This involves several critical steps:

1.  **Model Selection**:
    The service inspects the agent's `modelId` (e.g., `google:gemini-2.5-flash`, `openai:gpt-4o`) to select the appropriate underlying `ChatModel` bean (Google GenAI, OpenAI, or Anthropic).
    - *Fallback Strategy*: If no specific model is configured, it defaults to Google GenAI/Gemini.

2.  **System Prompt**:
    The agent's `description` is injected as the system prompt. For `COORDINATOR` teams, additional context about available team members is appended.

3.  **Advisors (RAG & Memory)**:
    Spring AI "Advisors" are attached to the client to intercept and augment the request:
    - **MessageChatMemoryAdvisor**: Loads conversation history from `ChatMemory` based on the `session_id`.
    - **QuestionAnswerAdvisor**: Retrieves relevant context from the `VectorStore` (PgVector) if RAG is enabled.
    - **PromptInjectionAdvisor / PIIAnonymizationAdvisor**: Applies security guardrails (if configured).

4.  **Tools**:
    The agent's tools (defined by name in `AgentDefinition`) are resolved from the `AgentRegistry` and attached to the client.
    - For **blocking calls**, `ToolCallAdvisor` is used to allow the LLM to autonomously invoke tools.
    - For **streaming calls**, tools are registered manually via callbacks (due to current streaming limitations with Advisors).

### C. Execution
Once the `ChatClient` is built, the prompt is executed:

```java
// Blocking
response = client.prompt()
    .user(userInput)
    .call() // Executes request & tool calls
    .content();

// Streaming
stream = client.prompt()
    .user(userInput)
    .stream() // Returns Flux<ChatResponse>
    .content();
```

Spring AI handles the interaction with the specific LLM provider API (e.g., Vertex AI, OpenAI), formatting the request and managing the network communication.

## 3. Result Handling

### Blocking Response
The final string response from the LLM (after potential tool execution loops) is wrapped in a `RunResponse` object containing:
- `content`: The assistant's text response.
- `metadata`: Model used, usage stats.
- `status`: "COMPLETED" or "PAUSED" (if human intervention is required).

### Streaming Response
For streaming, the service returns a `Flux<AgentStreamEvent>`. This reactive stream emits events for:
- `START`: Stream initialization.
- `CONTENT_DELTA`: Chunks of text as they are generated by the LLM.
- `STOP`: Stream completion.

This architecture allows for dynamic runtime behavior where agents can use different models, tools, and configurations while sharing a unified execution pipeline.
